# Architecture Overview

## Runtime & Rendering
- **Framework:** Next.js 15 (App Router), React 19, TypeScript
- **Styling:** Tailwind CSS with CSS variables (HSL tokens) in `app/globals.css`
- **Theme:** Dark by default (`<html data-theme="dark">`); light tokens defined but not default
- **Static sub-sites:** Served directly from `/public`, linked by Next.js pages

## UI System
- **Tokens:** Defined in `globals.css` as CSS variables: background/surfaces, border, text levels, accent scale, semantic colors (success, warning, error, info), radii, and shadows.
- **Utilities:** Custom utilities under `@layer utilities` (e.g., `.container-tight`, `.section`).
- **Components:** Button uses **class-variance-authority (CVA)** for size/variant; inputs and selects follow the same tokenized system.
- **Icons:** `lucide-react` throughout.

## Movie/TV Show Tracker: AI Model Selection
- **Shared model metadata:** `app/lib/aiModels.ts` is client-safe and holds available Gemini model IDs, display labels, defaults, validation helpers, and localStorage keys.
- **Gemini request utility:** `app/lib/aiConfig.ts` builds plain JSON Gemini requests for Edge routes without tools, grounding, URL context, code execution, file search, or function calling.
- **Defaults:** Classification/title expansion defaults to `gemini-3.1-flash-lite`; recommendations default to `gemini-2.5-flash`.
- **Recommendation eligibility:** `candidateShows` includes Watching, Planned, On Hold, and Completed shows only when relevant viewers marked `wouldRewatch` as `yes` or `maybe`, before applying watcher preference tiers.
- **Recommendation quick toggles:** `mood/page.tsx` applies client-side filters (No rewatches / Watching only / Animation only / Exclude animation) on top of `candidateShows` before building the prompt — deliberately kept to the highest-value filters rather than a full facet UI.

## Movie/TV Show Tracker: List/Row Views, Batch Update, Review Queue, Season Checks
- **View modes:** `page.tsx` toggles between the original card list and a dense `ShowRow` list (persisted to `localStorage` under `shows-view-mode`). `FilterBar` collapses status/type/vibe/watcher chips behind a "Filters" disclosure by default and always shows a sort row (`updated` / `score` / `alpha` / `seasons` / `incomplete`) — mirroring the same collapsible-disclosure pattern already used elsewhere in this tool (e.g. `ShowForm`'s watcher section).
- **Selection + batch AI update:** `page.tsx` owns select-mode state; `ShowCard`/`ShowRow` render a checkbox when active. `SelectionToolbar` provides select-all-visible and "AI update". `BatchUpdateModal` reuses the single-show classify call (extracted into `lib/classifyShow.ts`, also used by `ShowForm`) sequentially per selected show, updating title/type/vibes/description and tracking per-show success/failure so partial failures don't block the batch.
- **Review-missing workflow:** `lib/reviewCompleteness.ts` defines a review as complete when `story`/`characters`/`vibes`/`wouldRewatch` are all set (brain power is explicitly excluded — it's context-only per `compositeScore.ts`). `ReviewQueueModal` walks a per-user snapshot of incomplete ratable (completed/dropped/on_hold) shows one at a time with "Save & next".
- **Check for new seasons:** Show docs optionally persist `metadataSource`/`metadataSourceId` (the provider + ID the classify pipeline resolved) once a show has been classified. `POST /api/seasons` compares each eligible show's recorded season count against TMDb's `number_of_seasons`, falling back to a live high-confidence TMDb title search when no stored ID exists. **Limitation:** only `tv`/`cartoon` shows are covered — AniList/Jikan (the anime metadata sources) model each cour/season as a separate title, so there's no reliable single "season count" to diff for anime.

## Transcriber: Pipeline & Modules
- **Access:** Firebase Auth gated to the single admin account (`ADMIN_EMAIL` in `app/tools/trip-cost/firebaseConfig.ts`, reused via `app/tools/transcriber/lib/firebase.ts`). No Firestore — this tool has no persisted *run* state at all; a transcription run's segments/text live only in page-local React state for that tab. (Device-local preferences — the settings store and speaker profiles below — do persist across sessions in that browser's `localStorage`/IndexedDB, never server-side.) The sign-in screen is login-only (`AuthForm`'s `allowSignUp={false}`) — account creation isn't exposed here, since anyone other than the admin would just be signed up and immediately blocked. The Settings modal (see below) also surfaces this restriction directly in the UI as a read-only note.
- **Server-side auth:** `app/lib/verifyFirebaseAuth.ts` is a minimal, hand-rolled Firebase ID token verifier (RS256 signature check against Google's public JWKS via Web Crypto, plus iss/aud/exp claim checks) used because `firebase-admin` is Node-only and this app's API routes run on the Edge runtime. Every `/api/transcriber/*` route — including the three Gemini routes below — calls `requireAdminUser()` before doing any work; that call, not the client-side UI gating, is what actually protects the routes. `requireAdminUser` additionally requires the token's `email_verified` claim to be `true`; email/password sign-up on this shared Firebase project doesn't require proving address ownership up front, so without that check anyone who signed up with `ADMIN_EMAIL` (if not already registered) would pass the email-match check alone.
- **Requirements panel (`components/RequirementsPanel.tsx`):** Rendered at the top of the tool. Calls `GET /api/transcriber/status` with a freshly-forced ID token (`user.reload()` then `getIdToken(true)`) to explain, before a run, exactly which of `requireAdminUser`'s checks would fail and why. `GEMINI_API_KEY` now gates both the Gemini cleanup pass and the Gemini direct-transcription provider, so the status route's single `correctionKeyConfigured` field (and the panel's copy) covers both uses.
- **Settings store (`lib/settings.ts`):** A single versioned localStorage object, `transcriber_settings_v1` (`TranscriberSettings`, `version: 1`), replacing the tool's earlier two ad-hoc per-model keys. `parseStoredSettings(raw, legacy?)` is pure — never throws; every field is validated/clamped independently with a per-field fallback to `DEFAULT_TRANSCRIBER_SETTINGS` — with numeric clamps for merge gap (0.5–10s), cleanup chunk size (300–1800s), cleanup overlap (0–300s), and cleanup temperature (0–1). `readTranscriberSettings()`/`saveTranscriberSettings()` are the SSR-safe browser-facing wrappers; on first read with no v1 key present, they non-destructively migrate the two legacy per-model keys into a seeded v1 object (the legacy keys are left in place, just unused). The settings object covers provider choice + auto-fallback + fallback order, speaker-clip toggles (OpenAI and the experimental Gemini one), raw/cleaned output visibility, suppression, turn merging, cleanup (model/temperature/chunk/overlap/strict), argument tagging, and debug mode. Speaker profile metadata (name/notes) is deliberately kept in a *separate* key, `transcriber_speaker_profiles_v1` (`lib/speakerProfiles.ts`, same pure-parse pattern), so resetting settings to defaults never deletes a user's profiles.
- **Settings modal (`components/SettingsModal.tsx`):** Rebuilt on the settings store — grouped sections (provider & fallback, speaker clips, outputs, suppression, merging, cleanup, argument tagging, debug) with one-line descriptions and a Reset-to-recommended-defaults action, plus the same read-only admin-restriction note as before. The OpenAI transcription-model list is `app/lib/transcribeModels.ts` (models with segment-level timestamps only); the cleanup/Gemini-transcription model lists reuse the app-wide `AVAILABLE_GEMINI_MODELS` catalog (`app/lib/aiModels.ts`, same list/UI pattern as the Movie/TV Show Tracker's settings page), with Gemini direct transcription further restricted to the Flash-family subset `GEMINI_TRANSCRIBE_MODELS` (`gemini-2.5-flash-lite`, `gemini-2.5-flash`, `gemini-3.1-flash-lite`, `gemini-3.5-flash`; default `gemini-2.5-flash`).
- **Speaker profiles & reference clips (`useSpeakerProfiles.ts`, `components/SpeakerProfilesPanel.tsx`):** A collapsible panel above the upload form (auto-collapsed once every profile validates) supplies the speaker names/notes that used to be free-form upload-panel inputs. Profile metadata (name + voice notes) lives in `localStorage` (`lib/speakerProfiles.ts`); each profile's reference-clip **audio** lives in IndexedDB (`lib/speakerClips.ts` — db `transcriber`, store `speakerClips`, keyed by profile id), with an in-memory per-run fallback when `isClipStorageAvailable()` fails (a real open-and-close attempt, not just a feature check — Firefox private browsing exposes `indexedDB` but throws on open). A clip can be recorded (`useClipRecorder.ts`: MediaRecorder, `audio/webm;codecs=opus` with an `audio/mp4` Safari fallback, hard-stopped at 20s) or uploaded, then normalized by `lib/processReferenceClip.ts`: decode/downmix/resample to mono 16 kHz via the shared `lib/decodeAudioMono16k.ts` helper (also used by `geminiProvider.ts`'s per-window audio slicing below) → `lib/clipAnalysis.ts`'s `selectBestWindow` picks the highest-mean-RMS ~8s window → re-encode as a small PCM16 WAV. `computeClipValidation` reports `missing | too-short (<2s) | too-quiet (<-40 dBFS) | trimmed (>10s original) | ok` as a status pill; the panel also renders a copyable read-aloud script (`lib/voiceScript.ts`) for consistent recordings. `getRunClips()` resolves each profile's clip at submit time for whichever provider needs it.
- **Pipeline (`useTranscriberPipeline.ts`):** A client-orchestrated state machine: validate file extension/size (provider-aware cap — see Large files below) → duration probe (Gemini only, `lib/audioDuration.ts`) → provider attempt loop → capture raw output → suppression → chunked cleanup pass → turn merge → build outputs. The provider attempt loop tries `settings.provider` first; when `settings.autoFallback` is on (default off) and the classified error's `retryProviders` includes a not-yet-tried entry from `settings.fallbackOrder` (default `['gemini', 'openai-whisper']`), it hops to that provider once before surfacing the recovery panel — an explicit retry from the recovery panel never chains further auto-fallback. Raw segments are captured immediately after the first successful provider response, before suppression, so raw always preserves everything (no extra model call). `lib/runDebug.ts` accumulates a JSON-serializable, transcript-free debug log (provider attempts, raw segment count, suppression/cleanup counts, speaker-reference status, argument-tag summary, sanitized errors) in a ref; it only becomes visible state (`state.debugJson`) on failure or when `settings.debugMode === 'always'`.
- **Provider abstraction (`lib/providers/`):** `types.ts` defines `TranscriptionProviderId` (`'openai-diarized' | 'openai-whisper' | 'gemini'`), `TranscriptionAttempt`, and `TranscriptionAttemptError` (a plain classified-error object, not a thrown `Error`). `openaiProvider.ts` owns the XHR-with-upload-progress POST to `/api/transcriber/transcribe`; `geminiProvider.ts` owns the full Gemini flow. A recording at or under `GEMINI_SINGLE_CALL_MAX_SECONDS` uploads the original file once and transcribes it in a single call. A longer recording is never uploaded whole: `lib/decodeAudioMono16k.ts` decodes it once into mono 16 kHz samples, then each window's samples are sliced out, re-encoded as their own small WAV, and independently uploaded → polled → transcribed → best-effort deleted — so each `/gemini/window` call's audio input is exactly that window's slice, never the full recording. Both provider modules return a normalized `TranscriptionAttempt` or throw an already-`classifyTranscriptionError`-classified error, so the pipeline hook treats every provider uniformly.
- **Transcribe route (`app/api/transcriber/transcribe/route.ts`, OpenAI):** Re-validates the 25 MB upload cap server-side and resolves the model via `resolveTranscribeModelId` (default `gpt-4o-transcribe-diarize`); diarization-capable models use `response_format: diarized_json` + `chunking_strategy: auto`, others use `verbose_json` + segment timestamps (`lib/buildOpenAiTranscriptionForm.ts`'s `buildOpenAiTranscriptionEntries` makes the request shape unit-testable). The uploaded file's multipart content type is normalized before every OpenAI call via `resolveOpenAiUploadMime` — the extension-derived MIME wins over the browser-reported one for known audio extensions (OS MIME registries misreport real files, e.g. a valid `.m4a` arriving as `audio/mpeg`, which made OpenAI 400 the upload as "corrupted or unsupported"); the file is re-wrapped with the corrected type so the primary, known-speaker-retry, and whisper-fallback requests all send it. When a speaker profile has a clip and `settings.speakerClipsEnabled` is on, the client attaches clips as multipart `speakerClips[]` files plus a parallel `speakerClipNames` JSON array (≤4 clips, ≤5 MB each, `audio/*`, all re-validated server-side); the route base64-encodes each into a data URL and sends OpenAI's `known_speaker_names[]`/`known_speaker_references[]` fields (diarize model only). If OpenAI 400-rejects specifically the known-speaker fields, the route retries once on the same model without clips and adds `'speaker-references-rejected'` to the response's `warnings[]`. `lib/mapSpeakerLabels.ts` maps OpenAI's sequential speaker labels onto the provided speaker names in first-appearance order. The route's own single silent whisper-1 retry is now gated by an `allowWhisperFallback` field the client computes as `!isExplicitRetry && settings.autoFallback` — an explicit retry from the recovery panel never triggers it, and a fresh run only allows it when auto-fallback is on; with it off, a diarized-model failure surfaces to the client as a structured `{error, errorInfo}` instead of silently degrading to no-diarization.
- **Gemini direct transcription (`app/api/transcriber/gemini/{upload,file,window}/route.ts`):** A full alternative to the OpenAI path, client-orchestrated the same way the cleanup pass chunks a long transcript — a multi-hour file can't go through one `generateContent` call. `POST .../gemini/upload` proxies the file to Gemini's Files API (resumable-upload protocol; `GEMINI_API_KEY` never reaches the browser) and returns `{fileName, fileUri, mimeType, state}`. `GET .../gemini/file?name=…` polls activation (`GEMINI_FILE_POLL_INTERVAL_MS` 2s, `GEMINI_FILE_POLL_TIMEOUT_MS` 120s) until `state: 'ACTIVE'`; the same route's `DELETE` best-effort-deletes the file post-run (a failed delete is a warning, not an error). `POST .../gemini/window` builds one `generateContent` call (`lib/gemini/buildGeminiTranscriptionRequest.ts`) with `responseMimeType: 'application/json'` plus a structured `responseSchema` (`{segments: [{start, end, speaker, text}]}`, timestamps as strings since models emit `MM:SS`/`H:MM:SS` more reliably than floats) and parses the result (`lib/gemini/parseGeminiTranscription.ts`) — accepting number-seconds, `SS`, `MM:SS`, or `H:MM:SS(.mmm)`, applying a window-relative→absolute offset heuristic, clamping to window bounds, and normalizing speaker labels (exact profile-name match, or positional mapping for generic `Speaker 1`/`SPEAKER_00`-style labels, else `Unknown`). Recordings at or under `GEMINI_SINGLE_CALL_MAX_SECONDS` (20 min) go through in one call against the original uploaded file; longer recordings are windowed at `GEMINI_WINDOW_SECONDS`/`GEMINI_WINDOW_OVERLAP_SECONDS` (10 min / 15s) — but as **real sliced audio**, not a prompt-only restriction: `geminiProvider.ts` decodes the recording once client-side (`lib/decodeAudioMono16k.ts`), slices out each window's samples, and uploads/transcribes/deletes a separate small Gemini file per window, so a window call never resends (or gets billed for) the full recording. Windows reuse `createChunkWindows`/`stitchChunkResults` — the same core-region-filtering stitch as the OpenAI cleanup pass — and `buildGeminiTranscriptionRequest.ts`'s prompt tells the model each non-full-file clip is an excerpt starting at `windowStart`, so it reports absolute (not clip-relative) timestamps. Gemini has no acoustic speaker-reference support — speaker identity is prompt-inferred from names/notes/context notes; the default-off `settings.geminiReferenceClips` toggle additionally base64-encodes each profile's clip and attaches it as a labeled `inlineData` part (≤4 clips, ≤5 MB combined, re-validated server-side) as an extra, explicitly-unreliable signal — status/debug always label this "prompt-inferred" or "prompt-inferred+reference-clips (experimental)".
- **Cleanup pass (`app/api/transcriber/correct/route.ts`):** One Gemini call per overlapping chunk (`settings.cleanupChunkSeconds`/`cleanupOverlapSeconds`, default 15 min/90s; `settings.cleanupModel`, default `gemini-2.5-flash`) with a strict-JSON-out, preservation-first prompt (`lib/buildCorrectionPrompt.ts`) that fixes only obvious ASR errors, punctuation, and speaker-attribution flips — never summarizing, sanitizing, or inferring — and asks for `Unknown` when uncertain. When `settings.argumentTagging` is on (default off), the same call additionally asks for an optional per-segment `ArgumentTag` (`argument_conflict | repair_attempt | emotional_support | logistics_or_normal | unrelated | unclear`) — no separate AI pass; an invalid/missing tag falls back to `'unclear'` rather than rejecting the chunk. `temperature` is accepted from the client and server-clamped to `[0, 1]`. A syntactically valid but incomplete response (missing segment indices) rejects the whole chunk rather than silently patching gaps.
- **Suppression, merging, formatting (pure libs, run in this order after raw capture):** `lib/suppressArtifacts.ts` removes clusters of hallucinated short filler ("Hold on.", "Thank you.") from the *cleaned* path only (raw is unaffected) — a group of same-normalized-text, <4-word segments is removed only if it meets all three gates: repeat count (≥5 conservative / ≥4 aggressive), time span (≥90s), and gap regularity (coefficient of variation ≤0.4 conservative / ≤0.7 aggressive) — so scattered genuine short replies are never dropped. `lib/mergeTurns.ts` then merges consecutive same-speaker segments into `TurnBlock`s (gap under `settings.mergeGapSeconds`, never across a suppression boundary), carrying the majority tag when argument tagging is on (tag differences never block a merge). `lib/formatCleanTranscript.ts` renders one block per turn as `[HH:MM:SS] Speaker: text` with a blank line between blocks and the timestamp shown only once per block.
- **Error classification & recovery (`lib/classifyError.ts`, `components/ErrorRecoveryPanel.tsx`):** `classifyTranscriptionError({httpStatus, bodyText, provider, stage, ...})` is a pure heuristic classifier turning a raw failure into a stable `ErrorCategory` (quota, file-too-large, unsupported-format, auth, gemini-upload, gemini-parse, platform-limit, network, auth-config, unknown) plus user-facing `likelyCause`/`recommendedAction`/`retryProviders`; `suggestsConversion` is set only for `openai-unsupported-format`, never inferred from file size alone. When the rejected file's extension is one OpenAI supports (`.m4a`, `.mp3`, `.wav`, …), the `openai-unsupported-format` copy is file-specific — "OpenAI rejected this specific file's container/encoding, `.m4a` itself is supported" — never phrased as the format being unsupported, and a browser MIME that's implausible for the extension (e.g. `audio/mpeg` for `.m4a`) is called out as a hint the contents may not match the extension. `ErrorRecoveryPanel` replaces the old dead-end error message with a diagnostics table (provider/model, file name/size, browser vs. normalized MIME, sanitized upstream status/body) plus retry buttons (filtered by `retryProviders`, Gemini always offered), "Open settings", "Copy diagnostics", and — when a raw transcript already exists (transcription succeeded but cleanup failed) — "Download raw .txt" and "Complete with raw only".
- **Dual raw/cleaned outputs (`components/TranscriptOutput.tsx`):** Every successful run produces both a raw transcript (captured pre-suppression, always complete) and, when cleanup runs, a cleaned one (suppressed + corrected + turn-merged). Both are independently copyable/downloadable, raw collapsed by default; `settings.showRawOutput`/`showCleanedOutput` control visibility only, not whether raw is captured. When argument tagging produced output, an additional argument-relevant filtered export (`lib/argumentTags.ts`'s `filterArgumentRelevant` — core-tagged blocks plus `unclear`/untagged blocks sandwiched within `ARGUMENT_RELEVANT_GAP_SECONDS` of core-tagged neighbors on both sides) and a zero-filled tag-count summary are shown alongside.
- **Correction failures are tracked, not swallowed:** by default, a chunk whose correction call fails falls back to its uncorrected segments and the failure is counted; `lib/correctionSummary.ts` builds a "Completed with warnings: N of M correction chunks failed and were left uncorrected" message shown alongside the final transcript. An optional **strict correction** toggle aborts the entire cleanup pass on the first correction failure instead of silently falling back.
- **Skip cleanup:** a `ProviderPicker` toggle (`settings.cleanupEnabled: false`) that bypasses the Gemini cleanup pass entirely. `useTranscriberPipeline.ts` builds the raw transcript directly from the provider's segments and prepends `buildManualCleanupPrompt()`'s output — the same preservation-first rules as `buildCorrectionPrompt.ts`, in prose instead of strict-JSON-out, written for a human to paste (prompt + transcript together) into a general-purpose browser AI chat.
- **Chunking & stitching:** `lib/chunkTranscript.ts`'s `createChunkWindows` builds contiguous, non-overlapping "core" windows padded with overlap on both sides; `lib/stitchTranscript.ts`'s `stitchChunkResults` keeps only each window's core-range segments when reassembling (preventing overlap-region duplicates by construction, plus a belt-and-braces exact-match de-dup). Shared by both the cleanup pass and Gemini's windowed direct transcription, tuned with separate constants for each (`CORRECTION_CHUNK_SECONDS`/`CORRECTION_OVERLAP_SECONDS` vs. `GEMINI_WINDOW_SECONDS`/`GEMINI_WINDOW_OVERLAP_SECONDS`).
- **Large files:** OpenAI providers reject uploads over 25 MB (`MAX_OPENAI_UPLOAD_BYTES`), client- and server-side, with a message suggesting compression, splitting, or switching to Gemini. Gemini's Files API path accepts uploads up to 95 MB (`MAX_GEMINI_UPLOAD_BYTES`, kept under this deployment's Cloudflare Pages 100 MB request-body limit) — a real fallback for oversized files, not a documented gap. Accepted extensions are `.m4a, .mp3, .wav, .aac, .ogg, .webm` (`ACCEPTED_FILE_EXTENSIONS`), widened from the original `.m4a`-only support for coherence with the "convert to MP3/WAV/AAC" recovery guidance.
