# Architecture Overview

## Runtime & Rendering
- **Framework:** Next.js 15 (App Router), React 19, TypeScript
- **Styling:** Tailwind CSS with CSS variables (HSL tokens) in `app/globals.css`
- **Theme:** Dark by default (`<html data-theme="dark">`); light tokens defined but not default
- **Static sub-sites:** Served directly from `/public`, linked by Next.js pages

## UI System
- **Tokens:** Defined in `globals.css` as CSS variables: background/surfaces, border, text levels, accent scale, semantic colors (success, warning, error, info), radii, and shadows.
- **Utilities:** Custom utilities under `@layer utilities` (e.g., `.container-tight`, `.section`).
- **Components:** Button uses **class-variance-authority (CVA)** for size/variant; inputs and selects follow the same tokenized system.
- **Icons:** `lucide-react` throughout.

## Movie/TV Show Tracker: AI Model Selection
- **Shared model metadata:** `app/lib/aiModels.ts` is client-safe and holds available Gemini model IDs, display labels, defaults, validation helpers, and localStorage keys.
- **Gemini request utility:** `app/lib/aiConfig.ts` builds plain JSON Gemini requests for Edge routes without tools, grounding, URL context, code execution, file search, or function calling.
- **Defaults:** Classification/title expansion defaults to `gemini-3.1-flash-lite`; recommendations default to `gemini-2.5-flash`.
- **Recommendation eligibility:** `candidateShows` includes Watching, Planned, On Hold, and Completed shows only when relevant viewers marked `wouldRewatch` as `yes` or `maybe`, before applying watcher preference tiers.
- **Recommendation quick toggles:** `mood/page.tsx` applies client-side filters (No rewatches / Watching only / Animation only / Exclude animation) on top of `candidateShows` before building the prompt — deliberately kept to the highest-value filters rather than a full facet UI.

## Movie/TV Show Tracker: List/Row Views, Batch Update, Review Queue, Season Checks
- **View modes:** `page.tsx` toggles between the original card list and a dense `ShowRow` list (persisted to `localStorage` under `shows-view-mode`). `FilterBar` collapses status/type/vibe/watcher chips behind a "Filters" disclosure by default and always shows a sort row (`updated` / `score` / `alpha` / `seasons` / `incomplete`) — mirroring the same collapsible-disclosure pattern already used elsewhere in this tool (e.g. `ShowForm`'s watcher section).
- **Selection + batch AI update:** `page.tsx` owns select-mode state; `ShowCard`/`ShowRow` render a checkbox when active. `SelectionToolbar` provides select-all-visible and "AI update". `BatchUpdateModal` reuses the single-show classify call (extracted into `lib/classifyShow.ts`, also used by `ShowForm`) sequentially per selected show, updating title/type/vibes/description and tracking per-show success/failure so partial failures don't block the batch.
- **Review-missing workflow:** `lib/reviewCompleteness.ts` defines a review as complete when `story`/`characters`/`vibes`/`wouldRewatch` are all set (brain power is explicitly excluded — it's context-only per `compositeScore.ts`). `ReviewQueueModal` walks a per-user snapshot of incomplete ratable (completed/dropped/on_hold) shows one at a time with "Save & next".
- **Check for new seasons:** Show docs optionally persist `metadataSource`/`metadataSourceId` (the provider + ID the classify pipeline resolved) once a show has been classified. `POST /api/seasons` compares each eligible show's recorded season count against TMDb's `number_of_seasons`, falling back to a live high-confidence TMDb title search when no stored ID exists. **Limitation:** only `tv`/`cartoon` shows are covered — AniList/Jikan (the anime metadata sources) model each cour/season as a separate title, so there's no reliable single "season count" to diff for anime.

## Transcriber: Pipeline & Modules
- **Access:** Firebase Auth gated to the single admin account (`ADMIN_EMAIL` in `app/tools/trip-cost/firebaseConfig.ts`, reused via `app/tools/transcriber/lib/firebase.ts`). No Firestore — this tool has no persisted *run* state at all; a transcription run's segments/text live only in page-local React state for that tab. (Device-local preferences — the settings store and speaker profiles below — do persist across sessions in that browser's `localStorage`/IndexedDB, never server-side.) The sign-in screen is login-only (`AuthForm`'s `allowSignUp={false}`) — account creation isn't exposed here, since anyone other than the admin would just be signed up and immediately blocked. The Settings modal (see below) also surfaces this restriction directly in the UI as a read-only note.
- **Server-side auth:** `app/lib/verifyFirebaseAuth.ts` is a minimal, hand-rolled Firebase ID token verifier (RS256 signature check against Google's public JWKS via Web Crypto, plus iss/aud/exp claim checks) used because `firebase-admin` is Node-only and this app's API routes run on the Edge runtime. Every `/api/transcriber/*` route — including the three Gemini routes below — calls `requireAdminUser()` before doing any work; that call, not the client-side UI gating, is what actually protects the routes. `requireAdminUser` additionally requires the token's `email_verified` claim to be `true`; email/password sign-up on this shared Firebase project doesn't require proving address ownership up front, so without that check anyone who signed up with `ADMIN_EMAIL` (if not already registered) would pass the email-match check alone.
- **Requirements panel (`components/RequirementsPanel.tsx`):** Rendered at the top of the tool. Calls `GET /api/transcriber/status` with a freshly-forced ID token (`user.reload()` then `getIdToken(true)`) to explain, before a run, exactly which of `requireAdminUser`'s checks would fail and why. `GEMINI_API_KEY` now gates both the Gemini cleanup pass and the Gemini direct-transcription provider, so the status route's single `correctionKeyConfigured` field (and the panel's copy) covers both uses.
- **Settings store (`lib/settings.ts`):** A single versioned localStorage object, `transcriber_settings_v1` (`TranscriberSettings`, `version: 1`), replacing the tool's earlier two ad-hoc per-model keys. `parseStoredSettings(raw, legacy?)` is pure — never throws; every field is validated/clamped independently with a per-field fallback to `DEFAULT_TRANSCRIBER_SETTINGS` — with numeric clamps for merge gap (0.5–10s), cleanup chunk size (300–1800s), cleanup overlap (0–300s), cleanup temperature (0–1), the OpenAI preprocessing speed factor (1.0–1.5; **default now 1.0** — a previously saved value, including the old 1.2 default, is preserved), argument-range expansion (30–300s), and the two parallel-request counts (`openaiParallelChunks` 1–8, default 4; `cleanupParallelChunks` 1–12, default 6 — both integers, rounded then clamped). `readTranscriberSettings()`/`saveTranscriberSettings()` are the SSR-safe browser-facing wrappers; on first read with no v1 key present, they non-destructively migrate the two legacy per-model keys into a seeded v1 object (the legacy keys are left in place, just unused). The settings object covers provider choice + auto-fallback + fallback order, speaker-clip toggles (OpenAI and the experimental Gemini one), raw/cleaned output visibility + `showQualityDetails`, suppression, turn merging, cleanup (model/temperature/chunk/overlap/strict), targeted speaker repair (`speakerRepairEnabled`, default on; `speakerRepairModel`, default `gemini-2.5-flash-lite`), argument classification (`argumentTagging`, `argumentClassifierModel` default `gemini-2.5-flash-lite`, `argumentExpandSeconds` default 90), debug mode, and OpenAI long-recording preprocessing (`openaiPreprocessing`/`openaiSilenceRemoval`/`openaiSpeedFactor`, all additive v1 fields — see below). Speaker profile metadata (name/notes) is deliberately kept in a *separate* key, `transcriber_speaker_profiles_v1` (`lib/speakerProfiles.ts`, same pure-parse pattern), so resetting settings to defaults never deletes a user's profiles.
- **Settings modal (`components/SettingsModal.tsx`):** Rebuilt on the settings store — grouped sections (provider & fallback, speaker clips, OpenAI long recordings, outputs, suppression, merging, cleanup, argument tagging, debug) with one-line descriptions and a Reset-to-recommended-defaults action, plus the same read-only admin-restriction note as before. The OpenAI transcription-model list is `app/lib/transcribeModels.ts` (models with segment-level timestamps only); the cleanup/Gemini-transcription model lists reuse the app-wide `AVAILABLE_GEMINI_MODELS` catalog (`app/lib/aiModels.ts`, same list/UI pattern as the Movie/TV Show Tracker's settings page), with Gemini direct transcription further restricted to the Flash-family subset `GEMINI_TRANSCRIBE_MODELS` (`gemini-2.5-flash-lite`, `gemini-2.5-flash`, `gemini-3.1-flash-lite`, `gemini-3.5-flash`; default `gemini-2.5-flash`).
- **Speaker profiles & reference clips (`useSpeakerProfiles.ts`, `components/SpeakerProfilesPanel.tsx`):** A collapsible panel above the upload form (auto-collapsed once every profile validates) supplies the speaker names/notes that used to be free-form upload-panel inputs. Profile metadata (name + voice notes) lives in `localStorage` (`lib/speakerProfiles.ts`); each profile's reference-clip **audio** lives in IndexedDB (`lib/speakerClips.ts` — db `transcriber`, store `speakerClips`, keyed by profile id), with an in-memory per-run fallback when `isClipStorageAvailable()` fails (a real open-and-close attempt, not just a feature check — Firefox private browsing exposes `indexedDB` but throws on open). A clip can be recorded (`useClipRecorder.ts`: MediaRecorder, `audio/webm;codecs=opus` with an `audio/mp4` Safari fallback, hard-stopped at 20s) or uploaded, then normalized by `lib/processReferenceClip.ts`: decode/downmix/resample to mono 16 kHz via the shared `lib/decodeAudioMono16k.ts` helper (also used by `geminiProvider.ts`'s per-window audio slicing below) → `lib/clipAnalysis.ts`'s `selectBestWindow` picks the highest-mean-RMS ~8s window → re-encode as a small PCM16 WAV. `computeClipValidation` reports `missing | too-short (<2s) | too-quiet (<-40 dBFS) | trimmed (>10s original) | ok` as a status pill; the panel also renders a copyable read-aloud script (`lib/voiceScript.ts`) for consistent recordings. `getRunClips()` resolves each profile's clip at submit time for whichever provider needs it.
- **Pipeline (`useTranscriberPipeline.ts`):** A client-orchestrated state machine: validate file extension/size (provider-aware cap — see Large files below) → duration probe (Gemini always via `lib/audioDuration.ts`; OpenAI only when `settings.openaiPreprocessing` is on, since only that path needs it to decide chunking — a probe failure there just falls back to deciding on file size alone, unlike Gemini where an unknown duration is fatal) → provider attempt loop → capture raw output → **deterministic speaker reconciliation** (pure) → **quality gate** (pure) → **targeted speaker repair** (only when the gate triggers) → suppression → chunked sparse cleanup pass → identity-safe turn merge → **argument classification** (separate stage, only when enabled) → build outputs. The provider attempt loop tries `settings.provider` first; when `settings.autoFallback` is on (default off) and the classified error's `retryProviders` includes a not-yet-tried entry from `settings.fallbackOrder` (default `['gemini', 'openai-whisper']`), it hops to that provider once before surfacing the recovery panel — an explicit retry from the recovery panel never chains further auto-fallback. Raw segments are captured immediately after the first successful provider response, before suppression or reconciliation, so raw always preserves the provider's per-chunk output (no extra model call). **Resume caches:** the hook keeps keyed in-memory caches across runs — completed OpenAI transcription chunks (`lib/providers/openaiProvider.ts`'s `TranscribeChunkCache`), completed Gemini transcription windows (`lib/providers/geminiProvider.ts`'s `GeminiWindowCache`), the whole successful `TranscriptionAttempt`, successful speaker-repair batches, successfully corrected cleanup chunks, and classification windows — one independent cache per stage, so a failure partway through never costs the work already done and a downstream setting change never invalidates upstream work. Stage cache keys are built by the pure `lib/stageCacheKeys.ts` (tested invariants: pipeline-schema + prompt versions participate; argument settings never touch the cleanup/attempt keys; reference clips key by SHA-256 content hash via `lib/contentHash.ts` — source audio deliberately stays `name|size|lastModified`, documented there, since hashing a ~95 MB file into memory for a per-session cache key isn't worth the mobile-memory cost). A fresh submission clears all caches up front AND reads only happen on an explicit retry/resume, so stale results are never served. A chunk or attempt that silently degraded to Whisper (`mode: 'fallback'`) is additionally never reused by a diarized retry. `resume()`/`retryWith(provider)` reuse the caches when the keys match; `reclassify()` reruns ONLY the classification stage (+ pure range construction) with current settings; `reset()` clears everything. `lib/runDebug.ts` accumulates a JSON-serializable, transcript-free debug log (provider attempts, reconciliation/quality reports, sparse patch counts, per-stage token usage as reported by providers, sanitized errors) plus a **stage manifest** (`StageManifest`: pipeline schema version, mapping algorithm version, build commit when exposed, per-stage models, safe settings snapshot, expected/completed chunks, reference-clip hashes, quality metrics, patch counts, usage, fallback path, warning codes); it only becomes visible state (`state.debugJson`) on failure or when `settings.debugMode === 'always'`.
- **Speaker identity & reconciliation (`lib/mapSpeakerLabels.ts`, `lib/segmentProvenance.ts`, `lib/reconcileSpeakers.ts`):** Every segment carries additive provenance (`SegmentProvenance` in `lib/types.ts`): a stable id (`s<chunk>-<index>`, position-based, never derived from text/names), source chunk index, original provider label, stable chunk-local speaker identity, resolved/candidate speaker + confidence, mapping source, conflict flag, and user-confirmation flag. The chunk-local mapper keeps exact case-insensitive known-name matches, maps anonymous labels positionally onto *unclaimed* names, and — the key fix — preserves every extra anonymous label as a DISTINCT unresolved local identity displayed as `Speaker A/B/…`, never a shared generic `Unknown` (a response like `Kait, A, James, A, B` keeps `A` and `B` as their own identities instead of discarding them because both names were claimed). After all chunks combine, `reconcileSpeakers` runs once, purely and deterministically (never Gemini): exact names and user confirmations seed anchor evidence, audio-overlap matches union identities across chunks, boundary continuity adds candidate-band priors, and later-chunk positional guesses are demoted to priors (they're exactly the signal that swapped speakers at chunk boundaries). Confidence policy is centralized in `lib/constants.ts` (≥0.9 auto-assign; 0.7–0.89 candidate-but-displayed-unresolved; below unresolved; two anchors within 0.2 of each other = recorded conflict, left unresolved). The evidence-map interfaces are the intended attachment point for future acoustic embeddings — deliberately no ONNX/embedding dependency yet.
- **Quality gate + targeted speaker repair (`lib/speakerQuality.ts`, `lib/speakerRepair.ts`, `app/api/transcriber/speaker-repair/route.ts`):** `analyzeSpeakerQuality` is pure and text-free: named/unresolved word counts, per-5-minute-window unresolved percentages, longest unresolved run, label/identity inventories, mixed-label chunks, boundary identity changes, conflicts, Whisper usage, and a confidence distribution. Repair triggers when overall unresolved words >2%, any 5-minute window >10%, any unresolved run >30s, or any mapping conflict — surfaced as a one-line UI warning (details behind `settings.showQualityDetails`); the transcript stays downloadable regardless. The repair stage sends ONLY unresolved/low-confidence segments plus a few resolved neighbors per batch to the authenticated `speaker-repair` route (default `gemini-2.5-flash-lite`, one escalation pass on still-unresolved segments with `gemini-2.5-flash`), which returns sparse `{patches: [{segmentId, speaker, confidence}]}` — no transcript text ever comes back; ids are validated against the request targets and speakers against the known names server-side, re-validated client-side on application, applied only at ≥0.9 confidence, never over a user-confirmed assignment, and always preserving the prior display speaker in `repairedFrom`. A repair failure never invalidates the transcript.
- **Provider abstraction (`lib/providers/`):** `types.ts` defines `TranscriptionProviderId` (`'openai-diarized' | 'openai-whisper' | 'gemini'`), `TranscriptionAttempt` (with an optional `preprocessReport` — see below), and `TranscriptionAttemptError` (a plain classified-error object, not a thrown `Error`). `openaiProvider.ts` owns the XHR-with-upload-progress POST(s) to `/api/transcriber/transcribe`; `geminiProvider.ts` owns the full Gemini flow. A recording at or under `GEMINI_SINGLE_CALL_MAX_SECONDS` uploads the original file once and transcribes it in a single call. A longer recording is never uploaded whole: `lib/decodeAudioMono16k.ts` decodes it once into mono 16 kHz samples, then each window's samples are sliced out, re-encoded as their own small WAV, and independently uploaded → polled → transcribed → best-effort deleted — so each `/gemini/window` call's audio input is exactly that window's slice, never the full recording. Both provider modules return a normalized `TranscriptionAttempt` or throw an already-`classifyTranscriptionError`-classified error, so the pipeline hook treats every provider uniformly.
- **OpenAI long-recording preprocessing (`lib/preprocessAudioPlan.ts`, `lib/preprocessOpenAiAudio.ts`, `lib/providers/openaiProvider.ts`):** `gpt-4o-transcribe-diarize` caps audio at `OPENAI_MAX_AUDIO_SECONDS` (1500s) and OpenAI caps a single upload at `MAX_OPENAI_UPLOAD_BYTES` (25 MB) — both of which a long/large recording can exceed. When `settings.openaiPreprocessing` is on (default) and the probed duration exceeds `OPENAI_SINGLE_REQUEST_MAX_SECONDS` (20 min) or the file exceeds the 25 MB cap, `transcribeWithOpenAi` never sends a single request: it calls `preprocessForOpenAi()`, which decodes the file once (`lib/decodeAudioMono16k.ts`), detects and removes real silence (`detectKeptIntervals` — frame-by-frame RMS-in-dBFS below a threshold, a run only removed once it's long enough, shrunk by an edge pad on both sides so speech is never clipped), optionally speeds up the whole trimmed buffer via per-chunk `OfflineAudioContext` renders (`settings.openaiSpeedFactor`, **default now 1.0** — the pitch shift measurably hurts diarization/word accuracy, so speed-up is opt-in), plans chunk boundaries in the final (post-speed-up) time domain that stay under both a duration and a byte cap (`planChunks` — greedy, preferring a natural kept-interval boundary over a mid-speech hard cut), gives every chunk after the first `OPENAI_CHUNK_OVERLAP_SECONDS` (10s) of leading audio overlap (each `PlannedChunk` carries `encodeStart` alongside its core bounds; the core budget shrinks by the overlap so caps still hold), and encodes each chunk as its own small WAV. Each chunk goes through the SAME `/api/transcriber/transcribe` route as a small file — up to `settings.openaiParallelChunks` (default 4) in flight at once via `lib/concurrency.ts`'s `mapWithConcurrency` (allSettled semantics, deliberately not the shows tool's fail-fast helper of the same name) — with any speaker reference clips (Phase 4) speeded up by the same factor first (`applySpeedFactorToClip`, falling back to the original clip on error) so pitch-shifted chunk audio still matches the reference voices. Every chunk is always attempted even when one fails, and each success lands in the caller's `TranscribeChunkCache` before any failure is surfaced, so a retry re-runs only the failed chunks; upload progress is the mean of per-chunk fractions (reported only until the first chunk completes), and chunk progress is "N of M done", not a current-chunk position. `combineChunkResponses` remaps every segment's timestamps from chunk-local final time back to ORIGINAL-recording time (`createChunkTimeMapper`, inverting speed-up then silence-removal), deterministically deduplicates the overlap regions by core ownership (a segment belongs to the chunk whose core contains its start — no duplicate text can survive), matches the dropped overlap duplicates against their owners to recover cross-chunk speaker-identity links (`matchOverlapLinks`, fed into reconciliation), and stitches the chunks into one transcript — `mode` is `'diarized'` only if every chunk came back diarized, otherwise `'fallback'` with a mixed-mode warning when chunks disagreed. The Gemini windowed path recovers the same links from its 15s window overlaps (`collectWindowOverlapLinks`) before core-only stitching discards them. A small file (or preprocessing disabled) takes exactly the pre-existing single-request path — no decoding, no behavior change. The original file is decoded/chunked entirely client-side and never itself uploaded to the server in this path, so the pre-run size cap widens to `MAX_OPENAI_PREPROCESSED_UPLOAD_BYTES` (same value as Gemini's 95 MB cap) whenever preprocessing is enabled.
- **Transcribe route (`app/api/transcriber/transcribe/route.ts`, OpenAI):** Re-validates the 25 MB upload cap server-side and resolves the model via `resolveTranscribeModelId` (default `gpt-4o-transcribe-diarize`); diarization-capable models use `response_format: diarized_json` + `chunking_strategy: auto`, others use `verbose_json` + segment timestamps (`lib/buildOpenAiTranscriptionForm.ts`'s `buildOpenAiTranscriptionEntries` makes the request shape unit-testable). The uploaded file's multipart content type is normalized before every OpenAI call via `resolveOpenAiUploadMime` — the extension-derived MIME wins over the browser-reported one for known audio extensions (OS MIME registries misreport real files, e.g. a valid `.m4a` arriving as `audio/mpeg`, which made OpenAI 400 the upload as "corrupted or unsupported"); the file is re-wrapped with the corrected type so the primary, known-speaker-retry, and whisper-fallback requests all send it. When a speaker profile has a clip and `settings.speakerClipsEnabled` is on, the client attaches clips as multipart `speakerClips[]` files plus a parallel `speakerClipNames` JSON array (≤4 clips, ≤5 MB each, `audio/*`, all re-validated server-side); the route base64-encodes each into a data URL and sends OpenAI's `known_speaker_names[]`/`known_speaker_references[]` fields (diarize model only). If OpenAI 400-rejects specifically the known-speaker fields, the route retries once on the same model without clips and adds `'speaker-references-rejected'` to the response's `warnings[]`. `lib/mapSpeakerLabels.ts` maps OpenAI's speaker labels with full provenance — exact known-name matches kept, anonymous labels positionally onto unclaimed names, extras preserved as distinct `Speaker X` local identities (see Speaker identity & reconciliation above) — and the route passes provider-reported token usage through when present. The route's own single silent whisper-1 retry is now gated by an `allowWhisperFallback` field the client computes as `!isExplicitRetry && settings.autoFallback` — an explicit retry from the recovery panel never triggers it, and a fresh run only allows it when auto-fallback is on; with it off, a diarized-model failure surfaces to the client as a structured `{error, errorInfo}` instead of silently degrading to no-diarization.
- **Gemini direct transcription (`app/api/transcriber/gemini/{upload,file,window}/route.ts`):** A full alternative to the OpenAI path, client-orchestrated the same way the cleanup pass chunks a long transcript — a multi-hour file can't go through one `generateContent` call. `POST .../gemini/upload` proxies the file to Gemini's Files API (resumable-upload protocol; `GEMINI_API_KEY` never reaches the browser) and returns `{fileName, fileUri, mimeType, state}`. `GET .../gemini/file?name=…` polls activation (`GEMINI_FILE_POLL_INTERVAL_MS` 2s, `GEMINI_FILE_POLL_TIMEOUT_MS` 120s) until `state: 'ACTIVE'`; the same route's `DELETE` best-effort-deletes the file post-run (a failed delete is a warning, not an error). `POST .../gemini/window` builds one `generateContent` call (`lib/gemini/buildGeminiTranscriptionRequest.ts`) with `responseMimeType: 'application/json'` plus a structured `responseSchema` (`{segments: [{start, end, speaker, text}]}`, timestamps as strings since models emit `MM:SS`/`H:MM:SS` more reliably than floats) and parses the result (`lib/gemini/parseGeminiTranscription.ts`) — accepting number-seconds, `SS`, `MM:SS`, or `H:MM:SS(.mmm)`, applying a window-relative→absolute offset heuristic, clamping to window bounds, and normalizing speaker labels (exact profile-name match, or positional mapping for generic `Speaker 1`/`SPEAKER_00`-style labels, else `Unknown`). Recordings at or under `GEMINI_SINGLE_CALL_MAX_SECONDS` (20 min) go through in one call against the original uploaded file; longer recordings are windowed at `GEMINI_WINDOW_SECONDS`/`GEMINI_WINDOW_OVERLAP_SECONDS` (10 min / 15s) — but as **real sliced audio**, not a prompt-only restriction: `geminiProvider.ts` decodes the recording once client-side (`lib/decodeAudioMono16k.ts`), slices out each window's samples, and uploads/transcribes/deletes a separate small Gemini file per window, so a window call never resends (or gets billed for) the full recording. Windows reuse `createChunkWindows`/`stitchChunkResults` — the same core-region-filtering stitch as the OpenAI cleanup pass — and `buildGeminiTranscriptionRequest.ts`'s prompt tells the model each non-full-file clip is an excerpt starting at `windowStart`, so it reports absolute (not clip-relative) timestamps. Gemini has no acoustic speaker-reference support — speaker identity is prompt-inferred from names/notes/context notes; the default-off `settings.geminiReferenceClips` toggle additionally base64-encodes each profile's clip and attaches it as a labeled `inlineData` part (≤4 clips, ≤5 MB combined, re-validated server-side) as an extra, explicitly-unreliable signal — status/debug always label this "prompt-inferred" or "prompt-inferred+reference-clips (experimental)".
- **Cleanup pass — sparse text correction (`app/api/transcriber/correct/route.ts`):** One Gemini call per overlapping chunk (`settings.cleanupChunkSeconds`/`cleanupOverlapSeconds`, default 15 min/90s; `settings.cleanupModel`, default `gemini-2.5-flash`), with up to `settings.cleanupParallelChunks` (default 6) chunk requests in flight at once client-side; strict correction mode stops NEW chunks from starting after the first failure (in-flight ones finish and cache). Each window's segments go up with their stable ids; the model returns ONLY changed segments as `{patches: [{id, text}]}` (`lib/buildCorrectionPrompt.ts` + `lib/parseCorrectionResponse.ts`) — an omitted segment is unchanged by definition, an empty patch array is a valid answer, and unknown/duplicate ids are dropped server-side, so the model can never add, delete, split, merge, or reorder segments. Speakers are shown as context only — speaker fixes moved to the repair stage, argument tags to the classification stage. The response is constrained server-side with a Gemini structured-output `responseSchema`, the route makes up to 2 model attempts (one automatic re-ask on upstream failure or invalid JSON), and the per-patch divergence guardrail (`lib/correctionGuards.ts`) drops any patch whose trimmed length falls outside [0.5×, 2×] of the original (originals under 20 chars exempt; empty output always drops) — reported as `revertedPatches`. Patches are applied client-side onto the window's own segments (all provenance survives), stitching keeps only core-window results, and provider-reported token usage rides back on the response. `temperature` is accepted from the client and server-clamped to `[0, 1]`.
- **Suppression, merging, formatting (pure libs):** `lib/suppressArtifacts.ts` removes clusters of hallucinated short filler ("Hold on.", "Thank you.") from the *cleaned* path only (raw is unaffected) — a group of same-normalized-text, <4-word segments is removed only if it meets all three gates: repeat count (≥5 conservative / ≥4 aggressive), time span (≥90s), and gap regularity (coefficient of variation ≤0.4 conservative / ≤0.7 aggressive) — so scattered genuine short replies are never dropped. `lib/mergeTurns.ts` then merges consecutive segments into `TurnBlock`s **only when their speaker identity provably matches** (`mergeIdentityKey`: both resolved to the same global speaker, both unresolved sharing the same stable local identity, or both legacy/identity-less with equal display strings) — different unresolved local identities, resolved-vs-unresolved pairs, and different user-confirmed speakers never merge even when their display strings coincide; gap under `settings.mergeGapSeconds`, never across a suppression boundary; every block retains its constituent segment ids (`segmentIds`) and takes the first as its own stable id. `lib/formatCleanTranscript.ts` renders one block per turn as `[HH:MM:SS] Speaker: text` with a blank line between blocks and the timestamp shown only once per block.
- **Argument classification & range export (`lib/argumentClassify.ts`, `app/api/transcriber/classify/route.ts`, `lib/argumentTags.ts`):** A separate authenticated stage over conversational TURN BLOCKS (never tiny ASR fragments), run after the transcript is built and only when `settings.argumentTagging` is on. Blocks go up in overlapping contextual windows (40 blocks per window, 6 shared; exceptionally long turns split into parts that share the block id); the model (default `gemini-2.5-flash-lite`) returns ONLY `{classifications: [{blockId, tag, confidence}]}` — validated server-side (ids/tags/confidence) — and every window's votes aggregate deterministically (highest confidence, then tag priority, then first seen; never only the final window), deduplicated by block id, with coverage validated (uncovered blocks default to `unclear` plus a warning). Classification failure never invalidates the cleaned transcript. The argument-relevant export replaced the old "sandwich" filter with range construction (`buildArgumentRanges`): every `argument_conflict`/`repair_attempt`/`emotional_support` block anchors a range expanded `settings.argumentExpandSeconds` (default 90s) before and after, every intervening block is included regardless of tag, overlapping/adjacent ranges merge, ranges keep block + segment ids in chronological order, and isolated unrelated stretches outside every range are excluded. Changing argument settings never reruns transcription or cleanup — `reclassify()` reuses cached window classifications when the model is unchanged (an expansion-only change costs zero model calls).
- **Eval tooling (`scripts/transcriberEval.ts`, `npm run transcriber:eval -- <path>`):** Local-only harness (vite-node) that runs the deterministic stages (reconciliation + quality gate) over a segments JSON file and prints text-free metrics — unresolved rate, label changes, chunk-boundary changes, mixed-label chunks, long unresolved runs, confidence distribution, triggers, and classification coverage when blocks are supplied. Ships a synthetic committed fixture (`scripts/fixtures/transcriber-eval-sample.json`); private exports go in the gitignored `eval-data/`.
- **Error classification & recovery (`lib/classifyError.ts`, `components/ErrorRecoveryPanel.tsx`):** `classifyTranscriptionError({httpStatus, bodyText, provider, stage, ...})` is a pure heuristic classifier turning a raw failure into a stable `ErrorCategory` (quota, file-too-large, unsupported-format, auth, gemini-upload, gemini-parse, platform-limit, network, auth-config, unknown) plus user-facing `likelyCause`/`recommendedAction`/`retryProviders`; `suggestsConversion` is set only for `openai-unsupported-format`, never inferred from file size alone. When the rejected file's extension is one OpenAI supports (`.m4a`, `.mp3`, `.wav`, …), the `openai-unsupported-format` copy is file-specific — "OpenAI rejected this specific file's container/encoding, `.m4a` itself is supported" — never phrased as the format being unsupported, and a browser MIME that's implausible for the extension (e.g. `audio/mpeg` for `.m4a`) is called out as a hint the contents may not match the extension. `ErrorRecoveryPanel` replaces the old dead-end error message with a diagnostics table (provider/model, file name/size, browser vs. normalized MIME, sanitized upstream status/body) plus retry buttons (filtered by `retryProviders`, Gemini always offered), "Open settings", "Copy diagnostics", and — when a raw transcript already exists (transcription succeeded but cleanup failed) — "Download raw .txt" and "Complete with raw only". When the failed run left saved work behind (`RecoveryInfo.resume` — completed transcription chunks, or a cleanup failure where the whole transcription is cached), the panel leads with a primary "Resume run" button (the per-provider retries demote to secondary) and a line saying exactly what a resume reuses, e.g. "7 of 10 transcription chunks already succeeded and are saved".
- **Dual raw/cleaned outputs (`components/TranscriptOutput.tsx`):** Every successful run produces both a raw transcript (captured pre-suppression/pre-reconciliation, always complete) and, when cleanup runs, a cleaned one (reconciled + repaired + suppressed + corrected + turn-merged). Both are independently copyable/downloadable, raw collapsed by default; `settings.showRawOutput`/`showCleanedOutput` control visibility only, not whether raw is captured. When classification produced output, the argument-relevant range-based export and a zero-filled tag-count summary are shown alongside, plus a "Re-run tagging with current settings" action wired to `reclassify()`. A speaker-quality warning line appears when the gate triggered; `settings.showQualityDetails` adds the full text-free metric block.
- **Correction failures are tracked, not swallowed:** by default, a chunk whose correction call fails falls back to its uncorrected segments and the failure is counted; `lib/correctionSummary.ts` builds a "Completed with warnings: N of M correction chunks failed and were left uncorrected" message shown alongside the final transcript. An optional **strict correction** toggle aborts the entire cleanup pass on the first correction failure instead of silently falling back.
- **Skip cleanup:** a `ProviderPicker` toggle (`settings.cleanupEnabled: false`) that bypasses the Gemini cleanup pass entirely. `useTranscriberPipeline.ts` builds the raw transcript directly from the provider's segments and prepends `buildManualCleanupPrompt()`'s output — the same preservation-first rules as `buildCorrectionPrompt.ts`, in prose instead of strict-JSON-out, written for a human to paste (prompt + transcript together) into a general-purpose browser AI chat.
- **Chunking & stitching:** `lib/chunkTranscript.ts`'s `createChunkWindows` builds contiguous, non-overlapping "core" windows padded with overlap on both sides; `lib/stitchTranscript.ts`'s `stitchChunkResults` keeps only each window's core-range segments when reassembling (preventing overlap-region duplicates by construction, plus a belt-and-braces exact-match de-dup). Shared by both the cleanup pass and Gemini's windowed direct transcription, tuned with separate constants for each (`CORRECTION_CHUNK_SECONDS`/`CORRECTION_OVERLAP_SECONDS` vs. `GEMINI_WINDOW_SECONDS`/`GEMINI_WINDOW_OVERLAP_SECONDS`).
- **Large files:** with OpenAI preprocessing off, OpenAI providers reject uploads over 25 MB (`MAX_OPENAI_UPLOAD_BYTES`), client- and server-side, with a message suggesting compression, splitting, or switching to Gemini. With `settings.openaiPreprocessing` on (default), the client-side cap on an OpenAI run widens to `MAX_OPENAI_PREPROCESSED_UPLOAD_BYTES` (95 MB, same as Gemini's cap) since the original file is never itself uploaded to the server in that path — see OpenAI long-recording preprocessing above; the server's own 25 MB re-check still applies per-chunk, which the client-side chunk planner guarantees each chunk stays under. Gemini's Files API path accepts uploads up to 95 MB (`MAX_GEMINI_UPLOAD_BYTES`, kept under this deployment's Cloudflare Pages 100 MB request-body limit) — a real fallback for oversized files, not a documented gap. Accepted extensions are `.m4a, .mp3, .wav, .aac, .ogg, .webm` (`ACCEPTED_FILE_EXTENSIONS`), widened from the original `.m4a`-only support for coherence with the "convert to MP3/WAV/AAC" recovery guidance.

## Recipe Standardizer: Import, Scaling & Nutrition Hooks
- **No AI calls:** the tool ships a copyable ChatGPT conversion prompt (`lib/prompt.ts`); the user converts recipes in ChatGPT manually and pastes strict JSON back. Import is strict-JSON-only by design — `lib/schema.ts`'s pure `parseRecipeJson` never throws and reports exact-path errors (`ingredients[2].id`, broken `ingredientRefs`/`sectionId` references) plus non-fatal warnings. The prompt's JSON shape deliberately has no `shoppingList` (the tool derives shopping/grocery views from ingredients — `lib/shoppingList.ts` — so they can't drift after edits or scaling; a pasted `shoppingList` is still tolerated and ignored) and instead pushes the model toward accurate ingredient metadata: `sectionIds`, `primarySectionId`, `groceryCategory`, clean names, prep notes, and step `ingredientRefs`.
- **Ingredients hub:** the accordion nav is Overview / Ingredients / Prep / Cook / Scaling / JSON — there is no separate Shopping panel. The Ingredients panel owns a segmented mode control: **Recipe Workflow** (grouped by primary section, with editing + nutrition matching), **Shopping / Pantry** (consolidated checklist grouped by first-use section), and **Grocery Category** (same consolidated checklist by category). Checkbox ticks are shared across the two consolidated modes (ephemeral view state).
- **Data model (`lib/types.ts`):** grams are the source of truth (`quantityG`); `equivalent` is display-only text. Steps reference ingredients by id so an ingredient edit/substitution flows through every structured display (sections, step chips, shopping modes). Step *text* is free prose: renaming an ingredient opens `StepTextReviewModal`, which lists every step referencing the ingredient or mentioning the old name, prefilled with a safe word-boundary replacement (`lib/stepTextUpdate.ts`) for the user to confirm/adjust — instruction prose is never rewritten silently. Workflow sections are typed `prep | execution | combined`; the Prep view renders sections with prep steps, the Cook view renders sections with active steps.
- **Scaling (`lib/scaling.ts`):** servings / multiplier / target-final-weight / portions all reduce to one factor applied to a working copy at render time (actual weight beats estimated as the weight reference). Equivalent text is scaled only when its leading number parses (incl. fractions/unicode fractions), otherwise shown "at 1×". "Bake scale into recipe" rewrites baseline grams (clearing the measured `actualFinalWeightG`) and then routes through the save-choice flow.
- **Saving:** one Firestore doc per recipe at `artifacts/recipe-standardizer/users/{uid}/recipes/{recipeId}` (`{name, recipe, createdAt, updatedAt}`); first save creates the path, loads re-run the import validator. Any change to an already-saved recipe must go through `SaveChoiceModal` (update / save-as-new with a suggested "Name v2" / cancel) — no silent overwrites.
- **Nutrition Tracker integration path:** each ingredient carries a `nutritionLink` (`linked | likely | unlinked`, food item id + confidence). `lib/nutritionMatch.ts` name-matches against CalorieTracker saved foods (`artifacts/default-app-id/users/{uid}/foodItems` — the static site's fallback app id); only exact normalized names auto-link, token matches are "likely" and flagged for review, and unlinked ingredients never block saving. Computing recipe nutrition from linked items is a documented follow-up, not in v1.
